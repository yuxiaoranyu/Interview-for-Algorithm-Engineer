<h1 id="q-001">1. 排序算法应该从哪些维度分析？</h1>

排序算法不能只背时间复杂度，还要从五个维度分析：

1. **最好、最坏、平均时间复杂度**

   判断不同输入分布下的表现。

2. **空间复杂度**

   是否需要额外数组，是否为原地排序。

3. **稳定性**

   相同 key 的元素排序后是否保持原相对顺序。

4. **是否基于比较**

   比较排序下界是 $O(n\log n)$，线性排序需要数据范围条件。

5. **工程常数与缓存友好性**

   同样复杂度下，常数和内存访问模式会影响真实性能。

常见排序对比：

| 算法 | 平均时间 | 空间 | 稳定性 | 适合场景 |
| --- | --- | --- | --- | --- |
| 冒泡 | $O(n^2)$ | $O(1)$ | 稳定 | 教学，不推荐工程使用 |
| 插入 | $O(n^2)$ | $O(1)$ | 稳定 | 小数组、近乎有序数组 |
| 选择 | $O(n^2)$ | $O(1)$ | 不稳定 | 交换成本高但比较可接受 |
| 归并 | $O(n\log n)$ | $O(n)$ | 稳定 | 链表排序、外部排序 |
| 快排 | 平均 $O(n\log n)$ | $O(\log n)$ | 不稳定 | 通用内存排序 |
| 堆排 | $O(n\log n)$ | $O(1)$ | 不稳定 | Top K、优先队列 |
| 计数 / 桶 / 基数 | 近似 $O(n)$ | 依赖范围 | 可稳定 | 整数、小范围、分布均匀 |

<h2 id="q-002">面试问题：稳定排序、原地排序、比较排序分别是什么意思？</h2>

**稳定排序**：

如果两个元素 key 相同，排序后它们的相对顺序不变。

例子：用户列表先按注册时间排序，再按会员等级稳定排序，则同等级用户仍保持注册时间顺序。

**原地排序**：

额外空间复杂度为 $O(1)$ 或很小，不需要额外开辟和输入同规模的数组。

**比较排序**：

排序过程只通过元素之间的比较确定顺序，例如快排、归并、堆排。比较排序的复杂度下界是 $O(n\log n)$。

AIGC 场景：

- 对检索结果按分数排序，不太关注稳定性。
- 对同分候选按原始召回顺序保留时，需要稳定排序。
- 大规模日志排序或外部排序需要关注空间和 IO。
- token logits 排 Top K 时通常只需要前 K 个，不需要完整排序。

<h2 id="q-003">面试问题：冒泡、插入、选择排序如何对比？</h2>

| 算法 | 思想 | 稳定性 | 特点 |
| --- | --- | --- | --- |
| 冒泡排序 | 相邻元素反复交换 | 稳定 | 交换次数多，工程价值低 |
| 插入排序 | 将元素插入已排序区间 | 稳定 | 对小数组和近乎有序数据友好 |
| 选择排序 | 每次选择最小元素放到前面 | 不稳定 | 交换次数少，但比较次数固定多 |

三者平均复杂度都是 $O(n^2)$，但工程表现不同。

插入排序常用于：

- 小数组排序。
- 快排递归到小区间后的优化。
- 近乎有序数据。
- 在线插入少量新元素。

<h2 id="q-004">面试问题：为什么实际工程中插入排序常比冒泡排序更有价值？</h2>

虽然冒泡排序和插入排序平均时间复杂度都是 $O(n^2)$，但插入排序通常更实用。

原因：

- 冒泡排序交换操作多，每次交换通常需要多次赋值。
- 插入排序主要是元素移动，常数更小。
- 插入排序对近乎有序数据可接近 $O(n)$。
- 很多标准库排序在小区间会切换到插入排序。

AI 场景类比：

当候选集很小，例如 beam search 中局部候选数量不大，或者 rerank 后只需对少量候选微调顺序，简单插入排序可能比复杂排序更划算。

<h1 id="q-005">2. 归并排序、快速排序、堆排序如何对比？</h1>

三者都是高频排序算法：

| 算法 | 平均时间 | 最坏时间 | 空间 | 稳定性 | 核心思想 |
| --- | --- | --- | --- | --- | --- |
| 归并排序 | $O(n\log n)$ | $O(n\log n)$ | $O(n)$ | 稳定 | 分治 + 合并 |
| 快速排序 | $O(n\log n)$ | $O(n^2)$ | $O(\log n)$ | 不稳定 | pivot + 分区 |
| 堆排序 | $O(n\log n)$ | $O(n\log n)$ | $O(1)$ | 不稳定 | 堆结构维护最值 |

选择建议：

- 需要稳定性：归并排序。
- 通用内存排序：优化后的快排 / introsort。
- 需要 Top K 或优先队列：堆。
- 数据太大无法一次进内存：外部归并排序。

<h2 id="q-006">面试问题：归并排序为什么是分治思想的典型代表？</h2>

归并排序分两步：

1. **分**

   将数组递归拆成两半，直到每个子数组长度为 1。

2. **合**

   将两个有序子数组合并成一个更大的有序数组。

递推式：

$$
T(n) = 2T(n/2) + O(n)
$$

所以：

$$
T(n) = O(n\log n)
$$

优点：

- 稳定。
- 最坏时间复杂度稳定。
- 适合链表排序。
- 适合外部排序。

AI 场景：

- 分布式日志、训练样本、检索结果可以分片排序后归并。
- 多个有序候选列表合并时，可用归并思想。
- 多路归并常和堆结合，用于合并 K 个有序流。

<h2 id="q-007">面试问题：快速排序为什么平均快但最坏会退化？</h2>

快排核心是选择 pivot，并将数组分成小于 pivot 和大于 pivot 的两部分。

如果每次 pivot 都把数组均匀分成两半：

$$
T(n) = 2T(n/2) + O(n) = O(n\log n)
$$

如果每次 pivot 都选到最小或最大值，分区极不均衡：

$$
T(n) = T(n-1) + O(n) = O(n^2)
$$

优化方式：

- 随机选择 pivot。
- 三数取中。
- 小区间使用插入排序。
- 递归深度过深切换堆排序，即 introsort。
- 对重复元素多的数组使用三路快排。

面试中可以说：快排快是因为原地、缓存友好、常数小；退化风险要靠 pivot 策略和混合排序控制。

<h2 id="q-008">面试问题：堆排序和优先队列在 Top K 中如何使用？</h2>

堆是一种完全二叉树，通常用数组实现。

下标关系：

- 父节点：`(i - 1) // 2`。
- 左孩子：`2 * i + 1`。
- 右孩子：`2 * i + 2`。

Top K 场景：

- 求最大的 K 个元素：维护大小为 K 的小顶堆。
- 遍历新元素，如果大于堆顶，则替换堆顶并下沉。
- 总复杂度 $O(n\log K)$。

大模型场景：

- logits 中选 Top K token。
- RAG 检索结果取 Top K 文档。
- Beam Search 保留 Top K 序列。
- 调度系统按优先级处理请求。

注意：

- 如果 K 很小，堆非常合适。
- 如果需要完整排序，用堆不一定最快。
- GPU 上 Top K 会有专门 kernel，思想仍然是避免全量排序。

<h1 id="q-009">3. 桶排序、计数排序、基数排序什么时候能做到 $O(n)$？</h1>

线性排序不基于比较，因此可以突破比较排序 $O(n\log n)$ 下界，但需要数据满足特定条件。

| 算法 | 条件 | 例子 |
| --- | --- | --- |
| 桶排序 | 数据能均匀分到有序桶 | 分数、年龄、长度分桶 |
| 计数排序 | key 范围小且为非负整数 | 类别 id、评分、token 长度 |
| 基数排序 | 可拆成若干位，位数有限 | 定长字符串、手机号、整数 |

<h2 id="q-010">面试问题：计数排序为什么要求取值范围不能太大？</h2>

计数排序需要额外数组 `count[k]` 统计每个取值出现次数。

复杂度：

$$
O(n + k)
$$

其中 $k$ 是取值范围。如果 $k$ 远大于 $n$，计数数组会浪费大量空间。

适合：

- 年龄。
- 类别 id。
- 小范围整数分数。
- token 长度桶。
- 分类标签统计。

不适合：

- 浮点数。
- 超大范围整数。
- 稀疏 key。

AI 场景：

训练数据按 label id、长度、分桶采样时，如果范围很小，计数排序和桶统计非常高效。

<h2 id="q-011">面试问题：桶排序适合哪些 AI 数据预处理场景？</h2>

桶排序适合数据能自然划分到有序桶的场景。

AI 中常见：

- 按文本长度分桶，减少 padding。
- 按图像分辨率分桶，减少 resize 和 padding 成本。
- 按音频时长分桶，提升 batch 利用率。
- 按置信度分桶，做后处理阈值分析。
- 按 embedding 相似度区间分桶，做粗筛。

例如 NLP 训练中，样本长度差异大。如果随机组 batch，短句会被 padding 到长句长度，浪费显存。按长度分桶后，同一个 batch 内长度相近，可以显著减少 padding。

<h2 id="q-012">面试问题：排序在 AIGC 系统中有哪些真实应用？</h2>

排序在 AIGC 中无处不在：

- logits 排序或 Top K 筛选。
- Beam Search 候选序列排序。
- RAG 文档按相似度排序。
- reranker 对候选文档重排。
- 目标检测按置信度排序，再做 NMS。
- 数据集按长度分桶排序。
- 训练日志按时间排序。
- Agent 工具候选按置信度排序。

注意：真实系统不一定完整排序。很多情况下只需要 Top K，用堆、quickselect 或专用 Top K kernel 更合适。

<h1 id="q-013">4. 二分查找的本质是什么？</h1>

二分查找的本质是减治：每次根据条件排除一半不可能区间。

基础模板：

```python
def binary_search(nums, target):
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = left + (right - left) // 2
        if nums[mid] == target:
            return mid
        elif nums[mid] < target:
            left = mid + 1
        else:
            right = mid - 1
    return -1
```

复杂度：

- 时间：$O(\log n)$。
- 空间：$O(1)$。

在 42 亿数据中查找一个值，最多约 32 次比较，这就是 $O(\log n)$ 的威力。

<h2 id="q-014">面试问题：二分查找为什么要求随机访问和有序性？</h2>

二分查找需要两个条件：

1. **有序性**

   才能通过比较中间值排除一半区间。

2. **随机访问**

   才能 $O(1)$ 访问中间位置。

数组适合二分，因为下标访问是 $O(1)$。

链表不适合二分，因为找中点需要遍历，整体效率可能退化。即使链表有序，二分也不能发挥优势。

AI 场景：

- 有序阈值表可二分。
- 分桶边界可二分。
- 时间戳日志可二分定位。
- 但对向量相似度搜索，普通二分不适用，因为高维空间没有简单线性顺序。

<h2 id="q-015">面试问题：二分查找有哪些高频变体？</h2>

常见变体：

| 变体 | 目标 |
| --- | --- |
| 第一个等于 target | lower equal |
| 最后一个等于 target | upper equal |
| 第一个大于等于 target | lower_bound |
| 最后一个小于等于 target | upper_bound - 1 |
| 旋转数组搜索 | 有序性局部存在 |
| 二分答案 | 对答案空间二分 |

`lower_bound` 模板：

```python
def lower_bound(nums, target):
    left, right = 0, len(nums)
    while left < right:
        mid = left + (right - left) // 2
        if nums[mid] < target:
            left = mid + 1
        else:
            right = mid
    return left
```

高频坑：

- `left <= right` 和 `left < right` 混用。
- `mid` 溢出，C++ 中用 `left + (right - left) / 2`。
- 更新边界不收缩导致死循环。
- 没想清楚返回的是元素值还是插入位置。

<h2 id="q-016">面试问题：二分答案在 AI 工程中有哪些应用？</h2>

二分答案用于“答案具有单调性”的问题。

形式：

- 如果某个值 `x` 可行，那么更大/更小的值也可行。
- 可以在答案区间上二分。

AI 工程例子：

- 在显存限制下二分最大 batch size。
- 在延迟 SLA 下二分最大并发数。
- 在召回率约束下二分相似度阈值。
- 在压缩质量约束下二分量化 bit 或阈值。
- 在训练稳定性中搜索学习率区间。

面试中可以说：二分不只是查数组，也可以查“最小可行答案”或“最大可行答案”。

<h1 id="q-017">5. 哈希表为什么平均可以做到 $O(1)$ 查询？</h1>

哈希表通过哈希函数把 key 映射到数组下标：

$$
index = hash(key) \bmod capacity
$$

如果哈希函数分布均匀且冲突少，查询时可以近似直接定位，因此平均 $O(1)$。

哈希表核心概念：

- key。
- hash function。
- bucket / slot。
- hash collision。
- load factor。
- rehash。

AI 场景：

- tokenizer vocab：token string -> token id。
- metadata：doc_id -> document info。
- session store：session_id -> state。
- feature cache：feature key -> value。
- dedup：样本 hash -> 是否出现。

<h2 id="q-018">面试问题：如何解决哈希冲突？</h2>

常见方法：

1. **开放寻址法**

   冲突后继续寻找下一个空槽。

   - 线性探测。
   - 二次探测。
   - 双重哈希。

2. **链表法**

   同一个 bucket 中用链表或其他结构保存冲突元素。

对比：

| 方法 | 优点 | 缺点 |
| --- | --- | --- |
| 开放寻址 | 内存连续、缓存友好、易序列化 | 对装载因子敏感，删除麻烦 |
| 链表法 | 对装载因子不那么敏感 | 指针开销，缓存不友好 |
| 链表转树 | 冲突严重时性能更稳 | 实现复杂 |

Java HashMap 在冲突链较长时会转红黑树，这是工业级哈希表处理极端冲突的典型思路。

<h2 id="q-019">面试问题：如何设计工业级 HashMap？</h2>

工业级 HashMap 需要考虑：

1. **哈希函数**

   足够快，分布尽量均匀。

2. **装载因子**

   例如超过 0.75 后扩容。

3. **扩容和 rehash**

   一次性 rehash 可能造成延迟尖刺，可采用渐进式 rehash。

4. **冲突处理**

   链表法、开放寻址法、链表转树。

5. **安全性**

   防止恶意 key 导致哈希碰撞攻击。

6. **并发**

   分段锁、读写锁、无锁结构或 copy-on-write。

7. **缓存友好**

   尽量减少指针跳转和随机内存访问。

AI 系统中的注意点：

- 多租户缓存要隔离。
- key 中不能包含未脱敏敏感信息。
- 大规模 embedding cache 要考虑淘汰策略。
- 线上服务关注 P99，不只看平均 $O(1)$。

<h2 id="q-020">面试问题：跳表为什么能替代部分平衡树场景？</h2>

跳表是对有序链表的多级索引扩展。

核心思想：

- 底层是有序链表。
- 上层建立稀疏索引。
- 查找时从最高层开始，逐层向下缩小范围。

复杂度：

- 查询：期望 $O(\log n)$。
- 插入：期望 $O(\log n)$。
- 删除：期望 $O(\log n)$。
- 空间：$O(n)$。

优势：

- 实现比红黑树简单。
- 支持有序范围查询。
- 插入删除局部调整。
- 适合并发实现。

真实应用：

- Redis Sorted Set 使用跳表。
- 时间线、排行榜、分数排序。
- 流式有序数据维护。

AI 场景：

- 检索候选按分数动态维护。
- 在线评测排行榜。
- Agent 任务按优先级和时间排序。
- 近实时日志或事件流索引。

## 高频速记

1. 排序要从时间、空间、稳定性、是否原地、是否比较排序分析。
2. 插入排序对小数组和近乎有序数据更实用。
3. 归并排序稳定，快排平均快，堆排序适合 Top K。
4. Top K 不一定要全量排序，堆或选择算法更高效。
5. 计数排序要求 key 范围小，桶排序要求数据易分桶且分布相对均匀。
6. 二分查找本质是减治，要求有序性和随机访问。
7. 二分答案适合具有单调可行性的工程参数搜索。
8. 哈希表平均 $O(1)$，但要处理冲突、装载因子、rehash 和安全。
9. LRU 常用哈希表 + 双向链表。
10. 跳表用多级索引换取期望 $O(\log n)$ 查询，常用于有序集合。
