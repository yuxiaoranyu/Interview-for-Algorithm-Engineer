<h1 id="q-001">1. 字符串匹配为什么仍然是 AI 工程高频基础？</h1>

大模型时代，字符串匹配没有过时，反而更常见。

原因是 AIGC 系统不仅处理 Tensor，也处理大量文本、代码、日志、Prompt、工具调用参数和配置文件。

典型场景：

- Prompt 模板匹配。
- 敏感词过滤与安全策略。
- 日志检索和错误定位。
- 代码 Agent 的源码搜索、diff、patch。
- RAG 文档切片、关键词召回、倒排索引。
- Tokenizer 的前缀匹配。
- 约束解码中的合法 token 集合匹配。

字符串算法解决的是“在序列中快速查找模式”的问题。文本是字符序列，代码是字符序列，token ids 也是整数序列，因此很多字符串算法可迁移到 token 序列处理。

<h2 id="q-002">面试问题：BF、RK、BM、KMP 分别解决什么问题？</h2>

常见字符串匹配算法：

| 算法 | 核心思想 | 平均 / 典型特点 | 适合场景 |
| --- | --- | --- | --- |
| BF | 暴力逐位匹配 | 简单，最坏 $O(nm)$ | 小规模、教学 |
| RK | 用哈希比较子串 | 多模式或快速过滤 | 文档查重、滑窗匹配 |
| BM | 从模式串右侧匹配，利用跳跃规则 | 实际工程常很快 | 长文本中查找较长模式 |
| KMP | 利用前后缀信息避免回退主串 | 稳定 $O(n+m)$ | 需要最坏复杂度保证 |

其中 $n$ 是主串长度，$m$ 是模式串长度。

AIGC 场景：

- 代码搜索可先用简单字符串匹配快速定位候选文件。
- 安全过滤可用多模式匹配，而不是逐个敏感词暴力扫。
- RAG 中关键词召回一般不会只靠 BF，而会借助倒排索引、BM25 等结构。

面试时可强调：字符串匹配算法的目标是减少重复比较，本质是利用模式串自身结构或哈希信息加速搜索。

<h2 id="q-003">面试问题：KMP 的前缀函数或 next 数组本质是什么？</h2>

KMP 的核心是：当匹配失败时，主串指针不回退，模式串根据已匹配部分的前后缀信息移动。

前缀函数表示：

> 对于模式串每个位置，记录当前位置之前的子串中，最长相等真前缀和真后缀长度。

例如模式串中有一段已经匹配成功，但下一个字符失败，如果已匹配部分存在相同前后缀，就可以把模式串移动到后缀对应位置继续匹配。

复杂度：

- 构建前缀函数：$O(m)$。
- 匹配主串：$O(n)$。
- 总复杂度：$O(n + m)$。

AI 场景类比：

- 在长日志中稳定查找错误模式。
- 在 token 序列中查找固定触发片段。
- 在代码 Agent 中扫描函数签名或导入语句。

KMP 高频坑点：

- `next` 数组定义有不同版本，面试时要先说明自己的定义。
- KMP 不是让比较次数为 0，而是避免主串重复回退。
- KMP 对单模式串最坏复杂度稳定，但实际工程中多模式匹配更常用 AC 自动机。

<h2 id="q-004">面试问题：BM 算法的坏字符和好后缀思想是什么？</h2>

BM 算法从模式串右侧向左匹配，失配时尽可能多移动模式串。

两个核心规则：

1. **坏字符规则**

   如果某个主串字符和模式串对应位置不匹配，就根据该字符在模式串中最靠右的位置决定移动距离。

2. **好后缀规则**

   如果模式串尾部已经匹配了一段后缀，则寻找模式串中是否还有相同子串，或是否有后缀的后缀能匹配模式串前缀。

BM 的优势是实际文本中常能跳过大量字符，因此平均表现很好。

AI 工程中：

- 日志系统和源码搜索会使用更工程化的字符串搜索算法。
- 文档处理中的关键词定位可以借鉴 BM 跳跃思想。
- 当模式串较长、字符集较大时，BM 通常比简单暴力更有效。

面试中可以不手写完整 BM，但要能说清楚：BM 利用失配位置的信息向右跳，不像 KMP 那样主要利用前缀函数。

<h1 id="q-005">2. Trie 为什么适合前缀匹配？</h1>

Trie 又叫前缀树，用树路径表示字符串或 token 序列。

特点：

- 根到某个节点的路径表示一个前缀。
- 共享相同前缀的字符串共享节点。
- 查询一个字符串或前缀的复杂度是 $O(L)$，$L$ 为字符串长度。

适合场景：

- 字典查询。
- 前缀补全。
- 敏感词匹配。
- URL 路由。
- tokenizer 词表匹配。
- LLM prefix cache。

Trie 的空间开销可能较大，工程上常用压缩 Trie、Radix tree、Double-array Trie 等优化。

<h2 id="q-006">面试问题：Trie 在 tokenizer 和 prefix cache 中如何使用？</h2>

Tokenizer 场景：

很多 tokenizer 需要在文本中寻找最长可匹配子词。例如 WordPiece 或 BPE 词表可用 Trie 加速前缀匹配。

流程：

1. 从当前位置进入 Trie 根节点。
2. 沿字符或字节逐步向下匹配。
3. 记录最后一个合法 token 终点。
4. 选择最长匹配或按 tokenizer 规则切分。

Prefix cache 场景：

LLM 服务中，多个请求可能共享相同系统提示、few-shot 示例或长上下文前缀。可以把 token 前缀存成 Trie / Radix tree：

- 节点表示 token 前缀片段。
- 边表示追加 token。
- 节点挂载已计算的 KV cache block。
- 新请求到来时，沿 Trie 找最长公共前缀并复用缓存。

这样可以减少重复 prefill，提高吞吐并降低延迟。

面试回答可以概括为：Trie 的本质是把“重复前缀”从多个序列中抽出来共享，tokenizer 共享词表前缀，prefix cache 共享上下文前缀。

<h2 id="q-007">面试问题：AC 自动机如何实现多模式串匹配？</h2>

AC 自动机用于在主串中同时匹配多个模式串。

它由两部分组成：

1. **Trie**

   把所有模式串插入前缀树。

2. **失败指针 fail**

   当当前路径匹配失败时，跳转到最长可用后缀对应的节点，类似 KMP 的回退思想。

匹配流程：

1. 沿 Trie 读取主串字符。
2. 如果当前字符没有对应边，则沿 fail 指针回退。
3. 如果走到某个终止节点，说明匹配到一个或多个模式串。

复杂度：

- 构建：与模式串总长度相关。
- 匹配：接近 $O(n)$。

AIGC 应用：

- 敏感词过滤。
- Prompt 注入关键词检测。
- 输出安全策略扫描。
- 日志告警规则匹配。
- 批量代码模式检测。

注意：AC 自动机擅长精确多模式匹配，不擅长语义相似匹配。语义安全检测仍需要分类模型、embedding 或规则系统组合。

<h2 id="q-008">面试问题：敏感词过滤、约束解码和代码补全如何用到 Trie 类结构？</h2>

Trie 类结构常用于“当前前缀决定下一步候选”的场景。

敏感词过滤：

- 将敏感词集合构建成 Trie 或 AC 自动机。
- 扫描输入 / 输出文本。
- 命中后替换、拦截或进入二次模型审核。

约束解码：

- 如果模型输出必须符合某个 schema、枚举值或词表约束，可用 Trie 表示合法 token 序列。
- 每一步根据当前 Trie 节点限制可选 token。
- 适用于结构化 JSON、SQL、函数名、实体名生成。

代码补全：

- 项目中的函数名、变量名、路径名可以建前缀索引。
- 用户输入前缀后，快速返回候选。
- 与 embedding 检索和语义排序结合后，效果更稳。

面试中要补一句边界：Trie 解决的是符号级前缀约束，大模型解决的是语义生成，真实系统往往两者结合。

<h1 id="q-009">3. 海量数据去重和检索为什么需要高级索引结构？</h1>

当数据规模很小时，数组遍历和哈希表已经足够；当数据达到百万、千万、亿级，内存、延迟和 IO 成本会变成主要问题。

高级索引结构解决的问题：

- 快速判断元素是否存在。
- 快速过滤不可能候选。
- 支持关键词检索。
- 支持相似向量检索。
- 降低内存占用。
- 控制查询延迟。

AIGC 系统中常见：

- 训练数据去重。
- 网页抓取 URL 去重。
- RAG 文档检索。
- 向量数据库。
- Prompt / Response 安全扫描。
- embedding cache 命中判断。

<h2 id="q-010">面试问题：位图 Bitmap 适合解决什么问题？</h2>

Bitmap 用一个 bit 表示一个整数是否存在。

例如要表示 ID `x` 是否出现过，可以将第 `x` 位设置为 1。

优势：

- 空间极省。
- 查询和设置接近 $O(1)$。
- 可用位运算做集合交、并、差。

限制：

- 适合整数且范围已知。
- 如果 ID 空间非常稀疏，可能浪费空间。
- 不直接保存元素附加信息。

AI 场景：

- 样本 ID 去重。
- URL 或文档编号去重。
- 标注任务完成状态。
- 推荐系统中用户已看内容过滤。
- 数据集切分中快速判断 train / val / test 归属。

如果 ID 空间巨大且稀疏，可以考虑 Roaring Bitmap 这类压缩位图。

<h2 id="q-011">面试问题：布隆过滤器为什么会误判但不会漏判？</h2>

布隆过滤器用多个哈希函数把元素映射到位数组的多个位置。

插入元素：

1. 用 $k$ 个哈希函数计算位置。
2. 将这些位置设为 1。

查询元素：

- 如果任意一个位置为 0，元素一定不存在。
- 如果所有位置都为 1，元素可能存在。

为什么会误判？

不同元素可能把同一批位置都置为 1，所以一个未插入元素也可能被判断为存在。

为什么不会漏判？

只要元素插入过，它对应的所有位置一定被置为 1，除非发生删除或数据损坏。

AI 场景：

- 大规模训练样本去重的前置过滤。
- RAG 文档抓取时判断 URL 是否访问过。
- 缓存系统中避免查询不存在的 key 穿透后端存储。
- 日志流中快速判断事件是否可能出现过。

面试中要说明：布隆过滤器适合容忍少量误判、但不能容忍漏判的场景；如果需要删除，可以用计数布隆过滤器。

<h2 id="q-012">面试问题：倒排索引和 RAG 检索有什么关系？</h2>

倒排索引从“词项”映射到“包含该词项的文档列表”。

结构示例：

```text
token_a -> doc1, doc7, doc20
token_b -> doc3, doc7
token_c -> doc2, doc20
```

传统搜索引擎会在倒排索引基础上计算 TF-IDF、BM25 等相关性分数。

RAG 中倒排索引仍然有价值：

- 精确召回关键词、实体名、编号、API 名称。
- 弥补纯向量检索对数字、代码、专有名词不稳定的问题。
- 与向量检索做 hybrid search。
- 为 reranker 提供候选池。

典型 RAG 检索链路：

1. Query 改写或抽取关键词。
2. 倒排索引召回关键词相关文档。
3. 向量索引召回语义相关文档。
4. 合并去重。
5. reranker 重排。
6. 送入 LLM 生成答案。

面试金句：向量检索擅长语义相似，倒排索引擅长精确匹配；生产级 RAG 通常需要两者结合。

<h1 id="q-013">4. 向量索引为什么是 AIGC 检索系统的核心结构？</h1>

向量索引用于快速查找与 query embedding 最相似的候选向量，是 RAG、推荐、图像检索、多模态检索的核心。

如果有 $N$ 个文档向量，每个维度为 $d$，精确搜索需要计算 query 与所有向量的距离，复杂度约为 $O(Nd)$。当 $N$ 达到千万或亿级时，精确搜索延迟和成本很高。

向量索引的目标：

- 降低查询延迟。
- 控制内存占用。
- 在召回率和速度之间权衡。
- 支持增量插入、删除和过滤。

常见距离：

- 内积。
- 余弦相似度。
- 欧氏距离。

<h2 id="q-014">面试问题：精确 kNN 和 ANN 有什么区别？</h2>

精确 kNN：

- 计算 query 与所有向量的距离。
- 返回真正最近的 K 个。
- 召回率最高，但成本大。

ANN，Approximate Nearest Neighbor：

- 通过图、聚类、量化等索引加速。
- 返回近似最近邻。
- 用少量召回损失换取显著速度提升。

对比：

| 维度 | 精确 kNN | ANN |
| --- | --- | --- |
| 准确性 | 最高 | 近似 |
| 延迟 | 高 | 低 |
| 内存 | 依赖原始向量 | 可压缩 |
| 适合规模 | 小到中等 | 大规模 |
| RAG 常用性 | 小库可用 | 生产常用 |

面试中要强调：ANN 不是错误算法，而是工程权衡。RAG 最终效果还会经过 reranker 和 LLM，因此检索阶段常允许一定近似。

<h2 id="q-015">面试问题：HNSW、IVF、PQ 的核心思想是什么？</h2>

常见 ANN 思想：

| 方法 | 核心思想 | 优点 | 代价 |
| --- | --- | --- | --- |
| HNSW | 构建多层小世界图，从高层到低层贪心搜索 | 召回高、速度快 | 内存开销较大 |
| IVF | 先聚类成多个倒排桶，查询时只搜部分桶 | 可扩展、便于大规模 | 需要训练聚类中心 |
| PQ | 将向量分段量化，用码本压缩距离计算 | 省内存 | 有量化误差 |

简单理解：

- HNSW 像在向量空间建“高速公路图”。
- IVF 像先把向量分区，只查最可能的几个区域。
- PQ 像把高维向量压缩成短码，降低存储和计算成本。

RAG 工程中：

- 中小规模、追求召回：常用 HNSW。
- 超大规模、需要压缩和分片：IVF / PQ 更常见。
- 过滤条件很多时，需要考虑向量索引和标量过滤的结合方式。

<h2 id="q-016">面试问题：如何设计一个面向 RAG 的混合检索结构？</h2>

一个更稳的 RAG 检索结构通常不是单一向量索引，而是混合检索。

推荐结构：

1. **文档存储**

   保存原文、chunk、标题、来源、时间、权限、版本等元数据。

2. **倒排索引**

   用于关键词、实体、编号、代码符号召回。

3. **向量索引**

   用于语义召回，支持 ANN。

4. **哈希表 / KV 存储**

   保存 doc_id 到元数据、chunk 内容、embedding 的映射。

5. **过滤结构**

   使用 bitmap、布隆过滤器或标量索引做权限、时间、类型过滤。

6. **Top K 和重排**

   合并多路召回结果，用堆维护候选，再用 reranker 精排。

工程注意点：

- 文档更新后要处理索引一致性。
- 相同文档多版本要去重。
- 权限过滤必须在生成前严格执行。
- 召回不等于答案，必须评估最终生成质量。
- 向量库参数要通过召回率、延迟和成本共同调优。

<h1 id="q-017">5. 计算几何为什么会出现在 AI 视觉和自动驾驶面试中？</h1>

计算几何处理点、线、多边形、凸包、交并比、空间关系等问题，在视觉 AI 中非常常见。

典型场景：

- 目标检测框的 IoU 和 NMS。
- 分割 mask 的面积与边界。
- OCR 文本框排序。
- 数据标注中的多边形区域。
- 自动驾驶中的 BEV 感知、车道线、多边形区域。
- 3D 检测中的点云、包围盒和投影。

几何题看似传统算法，但本质是 AI 数据预处理、评估指标和后处理的重要基础。

<h2 id="q-018">面试问题：如何计算任意多边形面积？</h2>

常用鞋带公式。对于按顺时针或逆时针排列的多边形顶点：

$$(x_0, y_0), (x_1, y_1), \dots, (x_{n-1}, y_{n-1})$$

面积为：

$$
S = \frac{1}{2}\left|\sum_{i=0}^{n-1}(x_i y_{i+1} - x_{i+1} y_i)\right|
$$

其中下标按模 $n$ 处理，即 $x_n=x_0, y_n=y_0$。

代码示例：

```python
def polygon_area(points):
    n = len(points)
    s = 0
    for i in range(n):
        x1, y1 = points[i]
        x2, y2 = points[(i + 1) % n]
        s += x1 * y2 - x2 * y1
    return abs(s) / 2
```

AI 场景：

- 计算分割标注区域面积。
- 过滤过小的 mask 或 OCR 文本框。
- 计算多边形 IoU 的前置步骤。
- 自动驾驶中计算可行驶区域面积。

<h2 id="q-019">面试问题：如何判断点是否在多边形内？</h2>

常用射线法：

从待判断点向右发出一条射线，统计射线与多边形边的交点数量。

- 交点数为奇数：点在多边形内。
- 交点数为偶数：点在多边形外。

代码示例：

```python
def point_in_polygon(point, polygon):
    x, y = point
    inside = False
    n = len(polygon)
    for i in range(n):
        x1, y1 = polygon[i]
        x2, y2 = polygon[(i + 1) % n]
        cond = (y1 > y) != (y2 > y)
        if cond:
            x_cross = x1 + (y - y1) * (x2 - x1) / (y2 - y1)
            if x_cross > x:
                inside = not inside
    return inside
```

工程坑点：

- 点在边界上要单独处理。
- 顶点交点容易重复计数。
- 浮点误差需要设置 epsilon。

AI 场景：

- 判断目标中心点是否落在 ROI 内。
- 自动驾驶判断点云点是否位于车道区域。
- 图像标注工具判断鼠标点击是否选中多边形。
- 视觉后处理中过滤区域外检测框。

<h2 id="q-020">面试问题：极角排序和凸包有什么用？</h2>

极角排序是按点相对于某个基准点的角度排序，常用于构建凸包或做几何扫描。

凸包是包含一组点的最小凸多边形，可以理解为用橡皮筋包住所有点形成的边界。

常见凸包算法：

- Graham Scan。
- Andrew Monotonic Chain。

凸包用途：

- 提取点集外轮廓。
- 计算目标区域边界。
- 简化 mask 或点云投影。
- 判断点集空间分布。

AI 场景：

- OCR 中对倾斜文本点集求外接边界。
- 目标检测后处理里处理旋转框和多边形框。
- 自动驾驶 BEV 中根据点云投影估计障碍物轮廓。
- 图像分割标注中生成对象外轮廓。

面试中可补充：实际视觉系统还会大量使用 IoU、NMS、旋转框交集、多边形裁剪等几何算法，它们都是数据结构和算法在 AI 后处理中的直接应用。

## 高频速记

- 字符串匹配的核心是减少重复比较。
- KMP 利用前后缀信息，主串不回退。
- BM 从右往左匹配，利用坏字符和好后缀跳跃。
- Trie 适合前缀匹配，AC 自动机适合多模式匹配。
- Tokenizer、prefix cache、约束解码都可以用 Trie 类结构理解。
- Bitmap 适合整数集合，布隆过滤器适合海量集合的可能存在判断。
- 布隆过滤器会误判但不会漏判。
- 倒排索引擅长精确关键词召回，向量索引擅长语义召回。
- HNSW 是图索引，IVF 是聚类倒排，PQ 是向量量化压缩。
- 计算几何在检测、分割、OCR、自动驾驶后处理中非常常见。
