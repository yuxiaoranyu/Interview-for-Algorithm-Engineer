<h1 id="q-001">1. 树结构为什么是面试和 AI 系统中的高频基础？</h1>

树是一种层级结构，由节点和边组成，除根节点外每个节点通常只有一个父节点。树适合表达“从整体到局部”“从父任务到子任务”“从粗粒度到细粒度”的关系。

常见树结构：

| 结构 | 核心特点 | 高频场景 |
| --- | --- | --- |
| 普通树 | 一个节点可有多个子节点 | 文件目录、组织结构、任务树 |
| 二叉树 | 每个节点最多两个孩子 | 遍历、递归、表达式树 |
| 二叉搜索树 | 左子树小于根，右子树大于根 | 有序查询、范围查询 |
| 平衡树 | 控制树高接近 $O(\log n)$ | Map、Set、索引 |
| 堆 | 完全二叉树，父子满足堆序 | Top K、优先队列 |
| Trie | 按字符或 token 前缀建树 | tokenizer、前缀匹配 |

AI 场景中，树并不只用于刷题：

- 代码 Agent 会解析 AST，AST 本质是树。
- Prompt / 任务分解可以形成任务树。
- 决策树、随机森林、GBDT 是传统机器学习中的树模型。
- Prefix cache 可用 Trie / Radix tree 组织共享前缀。
- 层级分类、标签体系、菜单意图识别都依赖树状结构。

**面试金句：**

树的核心价值是表达层级关系；遍历树的核心价值是把层级问题转化为可执行的递归、栈或队列流程。

<h2 id="q-002">面试问题：二叉树、二叉搜索树和平衡树有什么区别？</h2>

二叉树只限制每个节点最多两个孩子，不限制节点值大小关系。

二叉搜索树在二叉树基础上增加有序性：

- 左子树所有节点值小于或等于根节点。
- 右子树所有节点值大于或等于根节点。
- 中序遍历可得到有序序列。

平衡树进一步控制树高，避免退化成链表。常见平衡树包括 AVL 树、红黑树、B 树、B+ 树。

复杂度对比：

| 操作 | 普通二叉树 | 普通 BST | 平衡 BST |
| --- | --- | --- | --- |
| 查找 | $O(n)$ | 平均 $O(\log n)$，最坏 $O(n)$ | $O(\log n)$ |
| 插入 | 依赖位置 | 平均 $O(\log n)$，最坏 $O(n)$ | $O(\log n)$ |
| 删除 | 依赖位置 | 平均 $O(\log n)$，最坏 $O(n)$ | $O(\log n)$ |

AI 工程类比：

- 如果只表达层级任务，用普通树即可。
- 如果需要按 key 有序查找，例如时间戳、分数、版本号，可以用有序树或跳表。
- 如果要支撑数据库、索引、日志检索，通常需要 B+ 树、LSM Tree 或跳表等工程化结构。

<h2 id="q-003">面试问题：二叉树前序、中序、后序遍历的递归模板是什么？</h2>

二叉树递归遍历的关键是：先明确当前节点要做什么，再决定访问根节点的时机。

前序遍历：根、左、右。

```python
def preorder(root):
    if not root:
        return
    visit(root)
    preorder(root.left)
    preorder(root.right)
```

中序遍历：左、根、右。

```python
def inorder(root):
    if not root:
        return
    inorder(root.left)
    visit(root)
    inorder(root.right)
```

后序遍历：左、右、根。

```python
def postorder(root):
    if not root:
        return
    postorder(root.left)
    postorder(root.right)
    visit(root)
```

使用场景：

| 遍历 | 适合问题 | AI / 工程例子 |
| --- | --- | --- |
| 前序 | 先处理父节点，再处理子节点 | 任务树展开、复制树、序列化 |
| 中序 | 二叉搜索树有序输出 | 有序索引扫描 |
| 后序 | 先处理子节点，再汇总父节点 | 计算目录大小、反向释放资源、计算图反传 |

面试中容易加分的说法：后序遍历特别适合“子问题结果决定父问题结果”的场景，动态规划和自动求导中的反向传播也有类似味道。

<h2 id="q-004">面试问题：层序遍历为什么通常用队列实现？</h2>

层序遍历按层从左到右访问节点，本质是 BFS。

队列满足先进先出，天然适合保存“下一批待访问节点”。

```python
from collections import deque

def level_order(root):
    if not root:
        return []
    ans = []
    q = deque([root])
    while q:
        level = []
        for _ in range(len(q)):
            node = q.popleft()
            level.append(node.val)
            if node.left:
                q.append(node.left)
            if node.right:
                q.append(node.right)
        ans.append(level)
    return ans
```

AI 场景：

- 多轮任务拆解中，先执行第一层子任务，再扩展下一层任务。
- 计算依赖图中，逐层调度没有依赖的节点。
- 层级分类中，先粗分类，再细分类。
- Tree-of-Thought / 搜索式推理中，按深度扩展候选路径。

层序遍历的重点不是代码模板，而是“用队列维护当前可处理前沿”。

<h2 id="q-005">面试问题：树结构在 AI 场景中有哪些真实应用？</h2>

树结构在 AI 中非常常见：

| AI 场景 | 树结构作用 |
| --- | --- |
| Tokenizer | BPE / WordPiece 可借助 Trie 做前缀匹配 |
| Prefix Cache | 用前缀树复用相同 prompt 的 KV 状态 |
| 代码 Agent | AST 表达源码结构，便于定位函数、类和语法块 |
| 规划 Agent | 将大目标拆成任务树 |
| 决策树 / GBDT | 用树模型做分类、回归和特征交互 |
| 层级标签 | 从一级类目逐层走到细分类目 |
| UI 自动化 | DOM 树表示页面结构 |

例如代码 Agent 修改一个函数时，不应该只做字符串搜索；更稳的方式是解析 AST，定位函数定义、调用关系和作用域，再生成 patch。

大模型推理中的 prefix cache 也很典型：多个请求共享同一段系统提示和上文时，Trie / Radix tree 可以把公共前缀组织起来，减少重复 KV 计算。

<h1 id="q-006">2. 堆为什么适合维护最值和 Top K？</h1>

堆是一种特殊的完全二叉树，满足堆序性质：

- 大顶堆：父节点值大于或等于子节点值。
- 小顶堆：父节点值小于或等于子节点值。

堆的优势：

- 插入元素：$O(\log n)$。
- 删除堆顶：$O(\log n)$。
- 查看最值：$O(1)$。
- 建堆：$O(n)$。

堆特别适合“只关心最大或最小的少量元素”的问题。例如：

- Top K 文档召回。
- Top K token 采样。
- Beam Search 保留得分最高的候选序列。
- 推理请求按优先级调度。
- 多路归并合并多个有序流。

<h2 id="q-007">面试问题：堆如何用数组表示？</h2>

堆是完全二叉树，因此可以不用指针，而是用数组紧凑存储。

如果数组下标从 0 开始：

- 父节点：`(i - 1) // 2`。
- 左孩子：`2 * i + 1`。
- 右孩子：`2 * i + 2`。

如果数组下标从 1 开始：

- 父节点：`i // 2`。
- 左孩子：`2 * i`。
- 右孩子：`2 * i + 1`。

工程中下标从 0 开始更常见，数学推导和手写堆有时用 1 开始更方便。

AI 场景中，堆常作为候选集维护结构，而不是直接做大规模矩阵计算。比如 RAG 检索时，从多个 shard 返回候选文档，可以用小顶堆维护全局 Top K。

<h2 id="q-008">面试问题：堆的插入、删除和建堆过程是什么？</h2>

堆的核心操作是上浮和下沉。

插入元素：

1. 将新元素放到数组末尾。
2. 与父节点比较。
3. 如果违反堆序，则交换并继续上浮。

删除堆顶：

1. 将堆顶和末尾元素交换。
2. 删除末尾元素。
3. 从根节点开始下沉，直到恢复堆序。

建堆：

1. 从最后一个非叶子节点开始。
2. 依次向前执行下沉。
3. 总复杂度是 $O(n)$，不是 $O(n\log n)$。

Python 小顶堆示例：

```python
import heapq

def top_k(nums, k):
    heap = []
    for x in nums:
        if len(heap) < k:
            heapq.heappush(heap, x)
        elif x > heap[0]:
            heapq.heapreplace(heap, x)
    return sorted(heap, reverse=True)
```

AI 面试中可结合：

- 采样前只保留概率最高的 K 个 token。
- 多路召回时合并不同检索器返回的候选。
- GPU 任务队列按 deadline、cost、priority 排序。

<h2 id="q-009">面试问题：堆排序、优先队列和 Beam Search 有什么关系？</h2>

堆排序是利用堆结构反复取最值来完成排序，复杂度为 $O(n\log n)$，空间复杂度可做到 $O(1)$，但不稳定。

优先队列是堆最常见的抽象接口：

- `push`：加入候选。
- `pop`：取出优先级最高或最低的元素。
- `peek`：查看当前最优元素。

Beam Search 是序列生成中的典型优先队列场景：

1. 每一步扩展当前 beam 中的候选序列。
2. 为新候选计算累计得分。
3. 从所有候选中保留 Top B 条。
4. 重复直到结束。

如果候选总数很大，用完整排序代价高；使用堆或 Top K 选择可以减少开销。

注意：真实大模型推理中，Top K / Top P 通常由高度优化的 GPU kernel 完成，但它背后的算法思想仍然是避免对整个 vocabulary 做完整排序。

<h1 id="q-010">3. 图结构解决什么问题？</h1>

图由顶点和边组成，适合表达多对多关系。

图的类型：

| 类型 | 特点 | 场景 |
| --- | --- | --- |
| 无向图 | 边没有方向 | 社交关系、相似关系 |
| 有向图 | 边有方向 | 依赖关系、调用关系 |
| 加权图 | 边有权重 | 路径成本、相似度 |
| DAG | 有向无环图 | 计算图、任务调度 |
| 二分图 | 顶点可分两组，边只跨组 | 匹配、推荐 |

AI 场景：

- 计算图：算子之间的依赖关系。
- 知识图谱：实体和关系。
- RAG 图检索：文档、实体、主题、引用关系。
- Agent Workflow：任务节点和执行依赖。
- GNN：直接在图结构上学习节点、边或整图表示。

<h2 id="q-011">面试问题：邻接矩阵和邻接表如何选择？</h2>

图的两种常见表示：

| 表示 | 空间复杂度 | 判断边是否存在 | 遍历邻居 | 适合场景 |
| --- | --- | --- | --- | --- |
| 邻接矩阵 | $O(V^2)$ | $O(1)$ | $O(V)$ | 稠密图、小图 |
| 邻接表 | $O(V + E)$ | 依赖实现 | $O(degree)$ | 稀疏图、大图 |

大多数真实 AI 系统中的图是稀疏图，例如知识图谱、文档引用图、任务依赖图，所以邻接表更常见。

例子：

- 知识图谱中每个实体只连接有限数量的关系。
- RAG 文档图中每个 chunk 只连接相似 chunk 或同源 chunk。
- 计算图中每个算子只依赖少数输入节点。

如果需要高频判断两个节点是否相连，可以在邻接表中嵌套哈希集合，用空间换查询速度。

<h2 id="q-012">面试问题：BFS 和 DFS 如何对比？</h2>

BFS 使用队列，按层扩展；DFS 使用递归或栈，一条路走到底再回溯。

| 维度 | BFS | DFS |
| --- | --- | --- |
| 数据结构 | 队列 | 栈 / 递归 |
| 访问方式 | 层序扩展 | 深入探索 |
| 最短路 | 无权图可求最短步数 | 不保证最短 |
| 空间 | 可能保存一整层 | 依赖深度 |
| 典型问题 | 最短路径、层级遍历 | 连通性、回溯、环检测 |

AI 场景：

- BFS：Tree-of-Thought 中按深度扩展多个候选思路。
- DFS：Agent 规划中深入执行一个子任务链。
- BFS：知识图谱中查找 k 跳邻居。
- DFS：代码依赖分析、拓扑排序、环检测。

面试回答要点：BFS 更适合“最短距离和层次”，DFS 更适合“路径探索和递归结构”。

<h2 id="q-013">面试问题：如何判断二分图？</h2>

二分图可以把顶点分成两个集合，所有边都连接不同集合内的点。

判断方法：BFS 或 DFS 染色。

算法流程：

1. 每个未访问节点作为新连通分量起点。
2. 给起点染色 0。
3. 遍历邻居，邻居染相反颜色。
4. 如果发现相邻节点颜色相同，则不是二分图。

```python
from collections import deque

def is_bipartite(graph):
    color = {}
    for start in graph:
        if start in color:
            continue
        color[start] = 0
        q = deque([start])
        while q:
            u = q.popleft()
            for v in graph[u]:
                if v not in color:
                    color[v] = 1 - color[u]
                    q.append(v)
                elif color[v] == color[u]:
                    return False
    return True
```

AI 场景：

- 推荐系统中，用户和物品可以构成二分图。
- 标注任务中，标注员和样本可构成二分图，用于任务分配。
- 多模态检索中，图像和文本也可构成跨模态二分关系。

二分图匹配还能用于资源分配，例如把 GPU 资源、任务类型、模型实例做约束匹配。

<h1 id="q-014">4. DAG 和拓扑排序为什么在 AI 工程中很重要？</h1>

DAG 是有向无环图，适合表达依赖关系。

典型场景：

- 深度学习计算图。
- 数据处理 pipeline。
- 模型训练工作流。
- Agent 多工具调用依赖。
- CI/CD 任务依赖。
- 多阶段 RAG 流程：解析、切片、embedding、索引、召回、重排、生成。

拓扑排序用于把 DAG 中的节点排成一个线性顺序，保证每个节点都在它依赖的节点之后执行。

如果图中存在环，则无法做合法拓扑排序。这在工程中意味着依赖设计有问题，例如任务互相等待、模块循环引用、工作流死锁。

<h2 id="q-015">面试问题：Kahn 算法如何实现拓扑排序？</h2>

Kahn 算法基于入度。

流程：

1. 统计每个节点的入度。
2. 将入度为 0 的节点加入队列。
3. 每次取出一个入度为 0 的节点加入结果。
4. 删除它的出边，并将邻居入度减 1。
5. 如果邻居入度变为 0，加入队列。
6. 如果最终结果节点数小于总节点数，说明有环。

```python
from collections import deque

def topo_sort(graph):
    indeg = {u: 0 for u in graph}
    for u in graph:
        for v in graph[u]:
            indeg.setdefault(v, 0)
            indeg[v] += 1

    q = deque([u for u in indeg if indeg[u] == 0])
    order = []
    while q:
        u = q.popleft()
        order.append(u)
        for v in graph.get(u, []):
            indeg[v] -= 1
            if indeg[v] == 0:
                q.append(v)

    return order if len(order) == len(indeg) else None
```

AI 工程应用：

- 按依赖顺序执行数据处理节点。
- 检测 Agent workflow 是否存在循环依赖。
- 编译计算图时确定算子执行顺序。
- 训练 pipeline 中避免下游任务提前执行。

<h2 id="q-016">面试问题：DFS 如何做拓扑排序和环检测？</h2>

DFS 拓扑排序通常使用三色标记：

- 0：未访问。
- 1：访问中。
- 2：已完成。

如果 DFS 时遇到访问中的节点，说明存在环。

```python
def topo_sort_dfs(graph):
    state = {}
    order = []

    def dfs(u):
        if state.get(u, 0) == 1:
            return False
        if state.get(u, 0) == 2:
            return True
        state[u] = 1
        for v in graph.get(u, []):
            if not dfs(v):
                return False
        state[u] = 2
        order.append(u)
        return True

    for u in graph:
        if state.get(u, 0) == 0:
            if not dfs(u):
                return None
    return order[::-1]
```

Kahn 算法更直观，适合调度；DFS 更适合递归依赖分析和环检测。

AI 场景：

- 代码 Agent 分析模块导入循环。
- 工作流引擎检测节点依赖环。
- 自动微分系统需要确定反向传播顺序。

<h2 id="q-017">面试问题：Dijkstra、A* 和最短路在 AI 系统中有哪些应用？</h2>

最短路问题是在图中寻找从起点到终点代价最小的路径。

| 算法 | 适用条件 | 特点 |
| --- | --- | --- |
| BFS | 无权图 | 求最少边数 |
| Dijkstra | 非负权图 | 求单源最短路 |
| Bellman-Ford | 可有负权边 | 可检测负环 |
| Floyd | 小规模全源最短路 | $O(n^3)$ |
| A* | 有启发函数 | 搜索更有方向性 |

AI 应用：

- 自动驾驶路径规划。
- 机器人导航。
- 知识图谱路径推理。
- RAG 中沿文档图扩展最相关路径。
- Agent 工具链中寻找成本最低的执行路径。

A* 的关键是启发函数 $h(n)$。如果 $h(n)$ 不高估真实距离，A* 可以保证最优；如果只是工程启发，可能更快但不一定最优。

面试金句：最短路算法不只用于地图，也用于任何“状态转移有代价”的问题。

<h1 id="q-018">5. 为什么说大模型系统越来越像图系统？</h1>

现代 AIGC 系统已经不只是单次模型调用，而是由模型、工具、检索、记忆、任务、评测、监控组成的复杂系统。这些组件之间存在依赖、引用、调用和反馈关系，本质上越来越像图。

典型图结构：

- Prompt 节点、工具节点、模型节点、检索节点组成 Agent workflow。
- 文档 chunk、实体、主题、引用组成 RAG 图。
- 算子和 Tensor 组成计算图。
- 用户、内容、标签组成推荐图。
- 代码文件、函数、类、调用关系组成代码图。

把系统看成图后，可以使用成熟图算法回答问题：

- 哪些节点依赖某个结果？
- 哪条路径成本最低？
- 是否存在循环依赖？
- 哪些文档是同一主题社区？
- 哪个节点是瓶颈？

<h2 id="q-019">面试问题：计算图、知识图谱、RAG 图检索和 Agent Workflow 如何对应到图算法？</h2>

| 场景 | 节点 | 边 | 常用算法 |
| --- | --- | --- | --- |
| 计算图 | 算子 / Tensor | 数据依赖 | 拓扑排序、反向遍历 |
| 知识图谱 | 实体 | 关系 | BFS、路径搜索、子图匹配 |
| RAG 图检索 | chunk / entity / topic | 相似、引用、包含 | k 跳扩展、社区发现 |
| Agent Workflow | 任务 / 工具 / 模型调用 | 执行依赖 | DAG 调度、环检测 |
| 代码理解 | 文件 / 函数 / 类 | 调用、导入 | 依赖分析、影响范围分析 |

实际面试中可以这样回答：

RAG 的第一代形态通常是“向量相似检索”；更复杂的系统会引入图结构，把文档、实体、章节、引用、时间线连接起来，让检索不仅靠 embedding 相似度，也能沿关系链补全上下文。

Agent workflow 也是如此。只用线性链路难以表达并行、条件分支和回滚；用 DAG 可以更自然地表达依赖和调度。

<h2 id="q-020">面试问题：图算法在模型部署和任务调度中如何落地？</h2>

模型部署和任务调度里，图算法常用于依赖管理和资源优化。

落地场景：

1. **计算图优化**

   编译器通过图优化做算子融合、常量折叠、内存复用和执行顺序安排。

2. **Pipeline 编排**

   数据清洗、特征构建、训练、评测、部署构成 DAG，拓扑排序决定执行顺序。

3. **推理服务调度**

   请求、模型实例、GPU、队列可以抽象成图或匹配问题，用于降低排队延迟。

4. **Agent 任务执行**

   多工具任务可以建成 DAG，先执行无依赖节点，再汇总结果。

5. **影响范围分析**

   修改一个模型、特征或数据源后，沿依赖图找到受影响的下游任务。

面试中可强调：图算法的价值不是写复杂代码，而是把系统依赖显式化，让调度、排障、优化都有结构可依。

## 高频速记

- 树表达层级关系，图表达多对多关系。
- 二叉树前序是根左右，中序是左根右，后序是左右根。
- 层序遍历用队列，本质是 BFS。
- BST 中序遍历有序，但普通 BST 最坏会退化成链表。
- 堆适合最值、Top K、优先队列、Beam Search。
- 邻接表适合稀疏图，邻接矩阵适合稠密图和快速判断边。
- BFS 适合层级和无权最短路，DFS 适合递归探索、回溯和环检测。
- 拓扑排序只适用于 DAG，可用于计算图、workflow 和 pipeline。
- Kahn 算法看入度，DFS 拓扑看三色标记。
- 二分图可用 BFS / DFS 染色判断。
