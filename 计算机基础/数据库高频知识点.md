<h2 id="1.数据库DBMS和数据库系统有什么区别？">1.数据库、DBMS和数据库系统有什么区别？</h2>

数据库（Database）是长期存储、有组织、可共享的数据集合。数据库管理系统（DBMS）是管理数据库的软件，例如MySQL、PostgreSQL、MongoDB、Redis等。数据库系统（DBS）则是由数据库、DBMS、应用程序、运维工具、权限体系、备份恢复机制和人员流程共同组成的完整系统。

面试中可以这样回答：

1. 数据库解决“数据怎么组织和存储”。
2. DBMS解决“数据怎么安全、高效、并发地访问”。
3. 数据库系统解决“业务如何围绕数据形成稳定服务”。

在AIGC和AI Agent系统中，数据库不只是业务表。它还承载用户会话、任务状态、工具调用记录、RAG文档元数据、embedding索引、审计日志、模型评测结果等。数据库能力越强，Agent系统越容易从demo走向可恢复、可追踪、可治理的生产系统。


<h2 id="2.关系型数据库Redis向量数据库对象存储在AIGC系统中如何分工？">2.关系型数据库、Redis、向量数据库、对象存储在AIGC系统中如何分工？</h2>

AIGC系统通常不是只用一种数据库，而是多种存储组合：

| 存储类型 | 适合存什么 | AI场景 |
| --- | --- | --- |
| MySQL/PostgreSQL | 结构化业务数据、事务数据 | 用户、订单、任务、权限、Agent运行状态 |
| Redis | 缓存、计数器、队列、分布式锁、短期状态 | 限流、会话缓存、热点Prompt、任务去重 |
| 向量数据库/向量索引 | embedding及相似度检索 | RAG召回、图文检索、长期记忆检索 |
| 对象存储 | 大文件和非结构化文件 | 图片、视频、音频、模型权重、数据集、生成结果 |
| 日志/分析型数据库 | 大规模事件和指标 | token用量、失败率、延迟、评测结果、用户行为分析 |

一个典型RAG Agent链路是：

1. 原始PDF、图片、音视频放对象存储。
2. 文档解析后的chunk、标题、权限、来源等元数据放关系型数据库。
3. chunk embedding写入向量数据库或带向量扩展的数据库。
4. 热点检索结果、session状态放Redis。
5. 工具调用、审计日志、评测指标写入日志系统。

面试金句：数据库选型不是“哪个数据库更强”，而是“哪类数据、哪种访问模式、哪种一致性要求，用哪种存储最合适”。


<h2 id="3.事务ACID隔离级别和MVCC在Agent任务系统中有什么价值？">3.事务ACID、隔离级别和MVCC在Agent任务系统中有什么价值？</h2>

事务是一个不可分割的数据库操作序列，要么全部成功，要么全部失败。ACID分别是：

| 特性 | 含义 | Agent场景 |
| --- | --- | --- |
| 原子性 | 要么全做，要么全不做 | 创建任务、扣费、写状态必须一起成功 |
| 一致性 | 事务前后满足约束 | 任务状态不能从失败跳回运行中 |
| 隔离性 | 并发事务互不干扰 | 多worker不能重复领取同一任务 |
| 持久性 | 提交后可靠保存 | Agent执行记录可恢复、可审计 |

隔离级别用于控制并发下的脏读、不可重复读、幻读等问题。工程中常用的是读已提交和可重复读。隔离级别越高，一致性越强，但并发性能可能越低。

MVCC（多版本并发控制）的核心思想是：读操作读取某个版本快照，写操作生成新版本，读写尽量不互相阻塞。它适合读多写少的业务，例如查询Agent任务状态、读取用户配置、查看RAG文档元数据。

在Agent任务系统中，常见写法是用数据库事务实现“领取任务”：

```sql
UPDATE tasks
SET status = 'running', worker_id = 'worker-1'
WHERE id = 1001 AND status = 'pending';
```

如果影响行数为1，说明领取成功；如果为0，说明已被其他worker领取。这比在应用层先查再改更安全。


<h2 id="4.数据库索引为什么常用B树哈希索引和向量索引有什么区别？">4.数据库索引为什么常用B+树？哈希索引和向量索引有什么区别？</h2>

B+树适合作为关系型数据库索引，因为它有三个优势：

1. 树高低，磁盘IO次数少。
2. 叶子节点有序，适合范围查询和排序。
3. 查询、插入、删除复杂度稳定，一般为 `O(log n)`。

哈希索引通过哈希函数定位桶，等值查询很快，但不适合范围查询和排序。例如 `user_id = 123` 可以很快，但 `created_at between ...` 不适合哈希索引。

向量索引用于近似最近邻搜索，目标不是精确匹配某个key，而是在高维向量空间中找到语义相似的TopK。常见索引包括HNSW、IVF、PQ等。

| 索引类型 | 主要查询 | 是否有序 | 典型场景 |
| --- | --- | --- | --- |
| B+树 | 等值、范围、排序 | 是 | 用户表、任务表、时间范围查询 |
| 哈希索引 | 等值查询 | 否 | 缓存、key-value查找 |
| 向量索引 | 相似度TopK | 否 | RAG、图片检索、embedding召回 |

RAG系统中常见组合是：先用metadata过滤权限、租户、时间范围，再用向量索引做语义TopK，最后用reranker重排。


<h2 id="5.Redis缓存分布式锁限流和会话状态如何用于AIGC服务？">5.Redis缓存、分布式锁、限流和会话状态如何用于AIGC服务？</h2>

Redis的核心价值是高性能内存数据结构。它在AIGC服务中常见用法包括：

1. 缓存：缓存模型配置、用户权限、热点RAG结果、重复Prompt结果。
2. 分布式锁：防止多个worker重复处理同一任务或重复构建同一索引。
3. 限流：按用户、IP、API key控制QPS和token预算。
4. 会话状态：保存短期对话上下文、流式请求状态、任务进度。
5. 队列：使用List、Stream等结构做轻量任务队列。

分布式锁必须设置过期时间，避免持锁进程崩溃后锁永远不释放：

```bash
SET task:1001:lock worker-1 NX EX 30
```

限流可以使用计数器：

```bash
INCR rate:user:123:minute
EXPIRE rate:user:123:minute 60
```

面试中要指出：Redis快，但不是万能数据库。强事务、复杂查询、长期持久化、审计数据不能只依赖Redis。生产系统通常把Redis作为缓存和协调层，把最终状态落到关系型数据库或日志系统。


<h2 id="6.向量数据库在RAG系统中如何设计？">6.向量数据库在RAG系统中如何设计？</h2>

RAG中的向量数据库设计通常包含四层：

1. 文档层：原始文档、来源、版本、权限、更新时间。
2. chunk层：切分后的文本片段、标题层级、页码、段落位置。
3. embedding层：chunk对应向量、模型版本、归一化方式。
4. 检索层：向量索引、metadata过滤、TopK、重排和召回日志。

核心设计点：

| 问题 | 设计建议 |
| --- | --- |
| chunk如何切 | 按语义边界切分，保留标题、页码、上下文窗口 |
| embedding如何管理版本 | 保存embedding模型名和版本，模型升级时可灰度重建索引 |
| 权限如何控制 | metadata中保存tenant、user、role、document_acl，检索前或检索时过滤 |
| TopK如何设置 | 初召回K可以偏大，再用reranker压缩到较小上下文 |
| 如何避免幻觉 | 返回证据片段和来源，让生成阶段受检索证据约束 |
| 如何评估 | 记录query、召回片段、点击/采纳、答案质量和失败原因 |

向量数据库不是RAG的全部。高质量RAG还需要文档解析、chunk策略、metadata治理、重排模型、上下文压缩、引用约束和评测闭环。


<h2 id="7.Agent系统中的长期记忆短期记忆和状态存储如何落库？">7.Agent系统中的长期记忆、短期记忆和状态存储如何落库？</h2>

Agent系统通常有三类数据：

| 数据类型 | 生命周期 | 存储方式 | 示例 |
| --- | --- | --- | --- |
| 短期记忆 | 单轮或短会话 | Redis、内存、会话表 | 当前对话上下文、临时工具结果 |
| 长期记忆 | 跨会话 | 关系型数据库 + 向量索引 | 用户偏好、项目背景、历史总结 |
| 运行状态 | 任务级 | 关系型数据库、日志系统 | plan、step、tool call、error、retry |

一个可恢复Agent运行表可以包含：

```sql
CREATE TABLE agent_runs (
  id BIGINT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  status VARCHAR(32) NOT NULL,
  current_step INT NOT NULL,
  input TEXT,
  output TEXT,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

工具调用记录可以单独建表，保存工具名、参数摘要、返回摘要、耗时、错误码和审计信息。这样做的价值是：失败后可以恢复，线上问题可以追踪，危险工具调用可以审计，评测系统可以复盘Agent决策链。

面试回答要注意边界：不是所有上下文都应该长期保存。隐私数据、敏感prompt、临时文件路径、访问token要最小化存储，并设置过期策略和权限控制。


<h2 id="8.海量日志文档TopK外排序和bitmap如何支撑RAG与评测？">8.海量日志、文档TopK、外排序和bitmap如何支撑RAG与评测？</h2>

海量数据处理在AI系统中非常常见。比如：统计最常失败的工具、找最常触发幻觉的query、分析最高频检索文档、构建评测集、去重网页语料。

### TopK

当数据量可以分块处理时，常见方法是：

1. 按hash切分到多个小文件或分区。
2. 每个分区用哈希表统计频次。
3. 每个分区维护小根堆得到局部TopK。
4. 汇总局部TopK后再次堆排序得到全局TopK。

在AIGC服务中，可用于统计高频prompt模板、高频错误、热门知识库文档。

### 外排序

当数据无法全部放入内存时，先分块排序，再多路归并。它适合大规模日志按时间、用户、query、耗时排序，也适合离线评测结果归并。

### bitmap

bitmap用bit表示元素是否存在，适合大规模整数集合的存在性判断、去重、交并集计算。例如用户ID去重、文档ID召回覆盖率、评测样本命中率。

面试金句：海量数据题在AI工程中不是抽象算法题，而是日志分析、语料治理、RAG召回评估和系统排障的底层能力。


<h2 id="9.数据库高可用备份恢复和数据一致性在AI服务中如何理解？">9.数据库高可用、备份恢复和数据一致性在AI服务中如何理解？</h2>

AI服务的数据故障通常比模型故障更难恢复。模型可以重新加载，状态、任务、日志、用户数据一旦丢失，影响会更大。

高可用常见手段：

1. 主从复制：读写分离，提高读取能力和容灾能力。
2. 自动故障转移：主库故障时切换到从库。
3. 分库分表：应对超大规模数据和高并发写入。
4. 多副本存储：避免单点磁盘或节点故障。

备份恢复常见手段：

1. 全量备份：定期备份完整数据。
2. 增量备份：只备份变化部分。
3. 日志恢复：通过binlog/WAL恢复到某个时间点。
4. 恢复演练：定期验证备份是否真的可用。

在Agent系统中，一致性尤其重要。例如任务状态、扣费记录、工具调用审计不能只写一半。对于生成结果文件，通常使用“数据库记录 + 对象存储文件”的组合，需要设计补偿机制：文件上传成功但数据库写入失败，或数据库写入成功但文件上传失败，都要能清理或重试。


<h2 id="10.数据库安全权限和多租户隔离在Agent工具调用中为什么重要？">10.数据库安全、权限和多租户隔离在Agent工具调用中为什么重要？</h2>

Agent具备工具调用能力后，数据库不再只是被后端服务访问，也可能被模型间接驱动访问。风险包括越权查询、SQL注入、敏感数据泄漏、跨租户召回、危险写操作等。

工程防护建议：

1. 最小权限：Agent使用只读账号或受限账号，不使用DBA账号。
2. 参数化查询：禁止拼接SQL字符串。
3. 白名单工具：只暴露经过审计的查询接口，不让模型自由执行任意SQL。
4. 多租户过滤：所有查询必须带tenant/user权限条件。
5. 审计日志：记录谁在什么时间通过什么工具访问了哪些数据。
6. 敏感字段脱敏：手机号、邮箱、密钥、身份证等字段不要直接进入prompt。
7. 写操作审批：涉及删除、扣费、发消息、改权限的操作要有二次确认或策略校验。

面试中可以总结：Agent时代的数据库安全不只是传统SQL注入防护，还要考虑模型把自然语言意图转成工具调用后的越权风险。数据权限必须在工具层和数据库层共同收口，不能只相信提示词约束。


<h2 id="11.数据库范式反范式和SQL优化在AI业务中如何取舍？">11.数据库范式、反范式和SQL优化在AI业务中如何取舍？</h2>

数据库范式用于减少数据冗余和更新异常。常见理解是：

1. 第一范式：字段不可再分，保证原子性。
2. 第二范式：非主属性完全依赖主键，避免部分依赖。
3. 第三范式：非主属性不传递依赖主键，避免传递依赖。

在AI业务中，范式适合核心状态和强一致数据，例如用户、订单、任务、权限、模型版本。反范式适合读性能和分析效率，例如把任务状态、模型名、用户等级、token统计冗余到日志或宽表中，减少多表join。

SQL优化常见方向：

| 问题 | 优化方式 |
| --- | --- |
| 慢查询 | 使用 `EXPLAIN` 查看执行计划，确认是否走索引 |
| 范围查询慢 | 为时间、状态、租户等高频过滤字段建立组合索引 |
| 深分页慢 | 用游标分页或基于id/时间的seek分页 |
| join过多 | 拆分冷热数据、适度反范式、缓存热点结果 |
| 写入压力大 | 批量写入、异步落库、分区表、日志型存储 |
| RAG metadata过滤慢 | 为tenant、document_id、权限、更新时间建立索引 |

Agent系统中尤其要关注“状态查询”和“工具调用日志”两类表。前者要求一致性和可恢复，后者要求写入吞吐和可分析。面试回答可以总结：范式保证正确性，反范式提升查询效率，真正的工程取舍取决于读写比例、一致性要求和数据生命周期。


<h2 id="12.WALbinlogcheckpoint和恢复机制为什么重要？">12.WAL、binlog、checkpoint和恢复机制为什么重要？</h2>

数据库恢复机制解决的问题是：系统宕机后，哪些事务应该保留，哪些事务应该回滚，如何尽快恢复到一致状态。

核心概念：

| 概念 | 含义 | 价值 |
| --- | --- | --- |
| WAL | Write-Ahead Logging，先写日志再写数据页 | 保证崩溃恢复能力 |
| binlog | 记录数据库变更逻辑或语句 | 主从复制、增量恢复、审计 |
| redo log | 重做日志 | 已提交事务在崩溃后可以重做 |
| undo log | 回滚日志 | 未提交事务可以回滚，支持MVCC |
| checkpoint | 将内存中的脏页逐步刷盘并记录恢复点 | 缩短崩溃恢复时间 |

在AIGC和Agent系统中，这些机制的意义非常直接：

1. 任务状态不能因为服务重启而丢失。
2. 工具调用审计日志需要可追溯。
3. 用户扣费、配额、token用量必须能恢复。
4. RAG知识库索引构建过程需要支持断点恢复。
5. 线上误操作需要通过备份和日志恢复到指定时间点。

面试金句：数据库可靠性不是“数据写进表里”这么简单，而是通过日志、刷盘、快照、复制和恢复演练，确保崩溃后仍能回到一致状态。
